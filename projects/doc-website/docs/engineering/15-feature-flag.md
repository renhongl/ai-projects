# 灰度发布、回滚与 Feature Flag

## 这是什么

这是前端交付治理能力的核心部分，用来降低上线风险。

## 三个核心概念

- 灰度发布：先让一部分用户使用新版本。
- 回滚：出现问题后快速恢复到旧版本。
- Feature Flag：用配置开关控制功能启停。

## 为什么重要

- 新功能可能带来线上风险。
- 不是所有问题都适合重新发版解决。
- 控制开关和灰度比“全量直接上”安全得多。

## 你需要掌握的能力

- 理解前端静态资源发布的风险。
- 设计基础回滚方案。
- 理解功能开关和版本发布的关系。

## 实践任务

- 为一个新功能设计开关控制方式。
- 写一份最小回滚预案。
- 画出一次灰度发布流程图。

看到你已经翻到这一页了，恭喜你，我们来到了前端工程化中非常硬核、但也极具价值的板块：**前端交付治理**。

如果说“多环境配置”解决的是代码在不同阶段该连哪个后端，那么 **灰度发布、回滚与 Feature Flag** 解决的就是：**“如何让新代码安全地上线，出事了能一秒止血，而不是引发灾难。”**

下面为你梳理前端落地这三件套的核心方案：

---

## 一、 Feature Flag（功能开关）

Feature Flag 的核心逻辑是：**代码已经全量上线了，但通过配置开关来决定“哪些人”在“什么时间”能看到这个功能。**

### 1. 为什么它能代替频繁的代码分支合入？

过去为了等一个功能上线，分支可能积压几周甚至几个月，最后合并时冲突到怀疑人生。使用 Feature Flag 后，新旧代码都在 `main` 分支，只是加了判断。

### 2. 代码层面的具体实现

- **硬编码静态开关（不推荐，只适合临时控制）：** 类似多环境里的 `.env` 变量，改完需要重新打包。
- **动态配置中心（标准做法）：** 前端在应用初始化时，或者通过定时轮询、WebSocket，从后端配置中心（如开源的 [GrowthBook](https://www.growthbook.io/)、[Unleash](https://www.getunleash.io/)，或自研系统）拉取开关状态。

```javascript
// 示例：动态功能开关判断
import { useFeatureFlag } from '@/hooks/useFeatureFlag';

function App() {
  // 从配置中心或 Context 获取当前用户是否开启了新版 UI 的开关
  const isNewUiEnabled = useFeatureFlag('NEW_COMPASS_UI_2026');

  return <div>{isNewUiEnabled ? <NewDashboard /> : <OldDashboard />}</div>;
}
```

## 二、 灰度发布（Grayscale Release）

前端是静态资源（HTML/JS/CSS），无法像后端那样轻松通过网关（如 Nginx）按照权重直接分发流量。前端灰度通常有两种主流模型：

### 1. 方案 A：HTML 动态分发（基于 Nginx / BFF）—— 最推荐

这是目前大厂最常用的方案。JS/CSS 正常打包传到 CDN，但 **`index.html` 不直接放在 CDN 上，而是由 Nginx 或后端 Node.js（BFF）动态返回**。

- **做法：** 1. 当用户请求 `example.com` 时，Nginx 拦截请求，解析用户的 `Cookie`（或 UserID / IP）。

2. 按照灰度算法（例如：`Hash(UserID) % 100 < 10` 代表 10% 灰度）。
3. 如果命中灰度，Nginx 返回 `index-v2.html`；没命中，返回 `index-v1.html`。

### 2. 方案 B：客户端运行时灰度（基于 Feature Flag）

把灰度做成 Feature Flag 的一种策略。

- **做法：** 全量用户拿到的都是同一套前端代码。代码运行时，向配置中心发送当前用户的 ID，配置中心返回 `"NEW_FEATURE": true`（针对 10% 的用户）或 `false`。

## 三、 前端回滚（Rollback）

很多人觉得回滚就是“在 CI/CD 平台上把上一个成功的 Task 重新点一下 Deploy”。但实际在复杂线上环境中，这往往不够快，甚至可能引发二次灾难。

### 1. 为什么不能只靠“重新部署”？

- 重新打包 + 构建 + 传输静态资源，通常需要 3~10 分钟。在这几分钟里，线上可能一直在报错。
- 如果涉及增量覆盖，CDN 缓存可能清理不及时。

### 2. 最快的回滚方案：指针切换

前端打包的静态资源（如 `app.a1b2c3.js`）应该**只增不删（不可变部署）**。

- **方案一（对应 HTML 动态分发）：** 回滚时，只需要在配置中心或者 Nginx 上，把指向 `index-v2.html` 的指针一键切换回 `index-v1.html`。**耗时只需 1 秒，瞬间止血。**
- **方案二（对应 Feature Flag）：**
  如果是新功能引发的线上 P0 级 Bug，不需要去回滚整个前端版本。直接去配置中心把 `NEW_COMPASS_UI_2026` 彻底关闭（OFF），用户刷新页面（或者动态感知到）就直接退回旧功能了。

## 四、 核心避坑与自测（回应你的知识清单）

1. **Feature Flag 的代码债务：** 功能全量上线 1~2 周且稳定后，**必须及时清理掉代码里的 `if (isNewUiEnabled)` 判断**。否则半年后，你的代码里会充斥着无数的“僵尸开关”，维护成本极高。
2. **数据不兼容回滚：** 如果新功能上线伴随了**后端数据库表结构的变更**（例如删除了某个旧字段），前端此时单独回滚到旧版可能会直接崩溃。所以发布和回滚方案必须前后端对齐，坚持“数据库变更改造要向前兼容”的原则。
