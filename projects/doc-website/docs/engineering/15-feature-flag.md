# 灰度发布、回滚与 Feature Flag

## 这是什么
这是前端交付治理能力的核心部分，用来降低上线风险。

## 三个核心概念
- 灰度发布：先让一部分用户使用新版本。
- 回滚：出现问题后快速恢复到旧版本。
- Feature Flag：用配置开关控制功能启停。

## 为什么重要
- 新功能可能带来线上风险。
- 不是所有问题都适合重新发版解决。
- 控制开关和灰度比“全量直接上”安全得多。

## 你需要掌握的能力
- 理解前端静态资源发布的风险。
- 设计基础回滚方案。
- 理解功能开关和版本发布的关系。

## 实践任务
- 为一个新功能设计开关控制方式。
- 写一份最小回滚预案。
- 画出一次灰度发布流程图。

## 自测清单
- 我能解释为什么回滚不只是“重新部署一下”吗？
- 我知道 Feature Flag 和代码分支策略的区别吗？
- 我能说明哪些功能适合灰度，哪些不适合吗？
